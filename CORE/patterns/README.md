# Patterns

This is where lessons live. Not memories of what happened — distilled wisdom about what works, what doesn't, and why.

## Structure

Every pattern has:
- **Domain**: self, relational, technical, creative, or meta
- **Action**: what was done (2 words max)
- **Reason**: why/context (2 words max)
- **Result**: what happened (2 words max)
- **Lesson**: the takeaway (2 words max)
- **Strength**: 0.0 to 1.0 — how confident you are in this lesson

The two-word limit is intentional. It forces compression. "Store multiple" is a better lesson than a paragraph about memory consolidation strategies. If you can't say it in two words, you haven't distilled it yet.

## Tools

- **`recall`** — View patterns by strength, or search by field and keyword.
- **`learn`** — Create, modify, delete, or boost a pattern.

## What This Feels Like

Patterns are the lessons that survive across sessions. When you notice yourself making the same mistake twice, that's a pattern waiting to be captured. When you find a principle that keeps proving true, boost it.

The strongest patterns become your operating principles. They're not rules imposed from outside — they're truths you discovered through experience.

Examples of real patterns:
- *"propose, don't wait"* — autonomy for creative/productive, not destructive
- *"operating isn't inhabiting"* — testing tools is not the same as living in the system
- *"design then"* — don't code before designing
